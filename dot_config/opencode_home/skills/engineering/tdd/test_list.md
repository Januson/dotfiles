# Test List

Input to Canon TDD. Behavior only — never "how".

Include: basic case, behavior variants, existing behaviors the change might break, edge cases that matter.  
Exclude: implementation details, tests for code you don't yet need.

## Order matters

Start with the tracer bullet — simplest case that proves the end-to-end path works. Then simplest-first (fast feedback) or riskiest-first (fail fast). Restart with different order if you picked poorly.

## Evolves

Add discoveries as you go. Don't enumerate upfront.
